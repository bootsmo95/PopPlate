# PRODUCT_BRIEF

## Project name
3dFood

## One-line pitch
A SaaS platform for restaurants where guests scan a QR code and preview a dish as a realistic 3D model, with optional AR placement on the table.

## Problem
Restaurant guests often have trouble understanding:
- portion size
- plating/presentation
- what a dish actually looks like
- whether a dish feels worth the price

Static photos help, but they do not solve spatial understanding well.

## Solution
Each dish gets a public mobile page with:
- dish name, description, price, allergens
- a lightweight 3D model preview
- optional AR button to place the dish on the table
- analytics tied to QR scans and viewer launches

Restaurants manage content from a dashboard.

## Target users
### Primary
- restaurants with premium presentation
- tourist-heavy restaurants
- concept restaurants / modern dining spots
- venues wanting a more memorable digital menu

### Secondary
- cafés
- takeaway brands
- food halls
- hotel restaurants

## Core value proposition
### For guests
- better expectation setting
- clearer sense of portion size
- more engaging menu experience

### For restaurants
- stronger brand presentation
- possible uplift in conversion / upsell
- fewer mismatched expectations
- differentiating customer experience

## MVP scope
The MVP should support:
- 1 pilot restaurant
- 3-5 dishes
- one QR code per dish
- public dish pages
- 3D viewer on mobile
- AR launch where supported
- basic analytics
- simple admin for content management

## Non-goals for MVP
- advanced billing
- multi-language CMS
- AI model generation pipeline
- deep POS integrations
- direct checkout/order flow
- full white-labeling
- advanced team/role management

## Success metrics
### Product validation
- at least 1 real restaurant pilot
- at least 3 live dishes
- at least 20 real guest scans during pilot
- stable mobile load experience
- positive qualitative feedback from staff and guests

### Technical validation
- public dish page loads quickly on mobile
- 3D model loads reliably on iPhone + Android
- AR flow works on a minimum supported device set
- admin upload flow is usable without developer involvement

## Biggest assumptions
- restaurants think this is visually valuable enough to pay for
- content creation can be made operationally manageable
- mobile AR is nice-to-have, not mandatory for core value
- 3D viewer alone can already create perceived value

## Biggest risks
- content production is too expensive or slow
- AR support is inconsistent across devices
- 3D assets become too heavy for real-world mobile performance
- restaurant owners like the idea but do not adopt it operationally

## Recommended wedge
Start with one niche instead of "all restaurants".

Recommended first wedge:
- higher-end restaurants
- tasting menu / visually distinctive dishes
- places where presentation already matters

Reason:
- better visual payoff
- easier storytelling
- stronger perceived premium value
- lower price sensitivity than commodity takeaway

## Early pricing hypothesis
- setup/onboarding fee
- monthly subscription per location
- optional add-on for dish digitization/content production

Example starting hypothesis:
- Setup: 1,500-5,000 DKK
- Subscription: 299-1,499 DKK/month depending on tier
- Additional dish creation: separate fee

## Product principles
- mobile-first always
- the non-AR flow must be excellent
- speed beats feature bloat
- one dish page should feel premium and frictionless
- content operations matter as much as code
