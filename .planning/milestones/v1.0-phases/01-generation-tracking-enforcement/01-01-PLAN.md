**server/api/user/usage.get.ts** -- Create new file following the profile.get.ts analog pattern (per D-06).

```typescript
import { requireAuth, getDbUser } from '../../utils/auth'
import { hasUnlimitedAccess } from '../../utils/access'
import { getTierLimits } from '../../utils/tiers'
import { getBillingCycleStart, getMonthlyGenerationCount } from '../../utils/generation-usage'

export default defineEventHandler(async (event) => {
  const { user } = await requireAuth(event)
  const dbUser = await getDbUser(user.id)

  if (!dbUser) {
    throw createError({ statusCode: 404, message: 'User not found' })
  }

  const anchorDate = dbUser.createdAt
  const cycleStart = getBillingCycleStart(anchorDate)
  const used = await getMonthlyGenerationCount(user.id, cycleStart)
  const limits = getTierLimits(user.accountTier)
  const unlimited = hasUnlimitedAccess(user)

  return {
    used,
    limit: unlimited ? null : limits.maxGenerationsPerMonth,
    tierName: user.accountTier,
    cycleStart: cycleStart.toISOString(),
    unlimited,
  }
})
```

Return `limit: null` for admin users (unlimited access) so the frontend can distinguish "no cap" from a numeric cap. Return `tierName` so the frontend can display "free", "basic", or "pro" alongside usage.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> generate.post.ts | Untrusted user triggers generation; must enforce both monthly and per-dish limits server-side |
| client -> usage.get.ts | Authenticated user requests own usage data; must only return data for the session user |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | generate.post.ts | mitigate | Monthly limit check runs server-side before job insert; client cannot bypass. Both hasUnlimitedAccess guard and count query use server-side session user ID. |
| T-01-02 | Elevation of Privilege | usage.get.ts | mitigate | requireAuth(event) enforces authentication. getDbUser uses session user.id, not a client-provided ID. No user can request another user's usage. |
| T-01-03 | Information Disclosure | usage.get.ts | accept | Endpoint returns the user's own usage count and tier limit. This is non-sensitive data intended for the user. |
| T-01-04 | Denial of Service | generate.post.ts | mitigate | Monthly limit cap (15/50/150) inherently rate-limits generation job creation per user. Existing in-flight job check (409) prevents duplicate jobs. |
| T-01-05 | Spoofing | getBillingCycleStart | mitigate | Anchor date comes from dbUser.createdAt (server-side DB value), not from client input. Client cannot manipulate their cycle start. |
</threat_model>
